# User Stories: Lobby Screen

---

## US-1: View the Lobby on App Open

**As a** casual player,  
**I want** to see an attractive landing page when I open the application,  
**so that** I feel welcomed and understand immediately that I am about to play a card game.

### Acceptance Criteria

- [ ] When the application is opened (root URL), the Lobby screen is displayed — no redirect, no splash, no other screen is shown first.
- [ ] The screen displays the game title "La Escoba" as a prominent heading.
- [ ] The screen includes at least one decorative visual element that communicates the card-game or broom theme.
- [ ] The branding area contains no interactive controls.
- [ ] The screen is visually appealing and uses a playful, colorful aesthetic.
- [ ] The screen renders correctly on a 320 px wide mobile viewport without horizontal overflow.
- [ ] The screen renders correctly on a desktop viewport (1280 px wide) without layout breakage.

---

## US-2: Choose Game Mode

**As a** player on the Lobby screen,  
**I want** to select between Single Player and Local Multiplayer modes,  
**so that** the setup panel adapts to show the right configuration fields for my intended session.

### Acceptance Criteria

- [ ] The setup panel displays exactly two game mode options: "Single Player" and "Multiplayer".
- [ ] "Single Player" is selected by default when the Lobby first loads.
- [ ] Selecting a mode immediately updates the setup panel — relevant fields for the new mode appear and fields from the previous mode disappear.
- [ ] The mode selector never allows an empty/de-selected state — one option is always active.
- [ ] Switching from Single Player to Multiplayer hides the AI difficulty selector.
- [ ] Switching from Multiplayer to Single Player hides the player-count selector.

---

## US-3: Configure a Single Player Session

**As a** player who wants to play against the AI,  
**I want** to set my name and choose the AI's difficulty,  
**so that** the game engine knows who I am and how challenging the opponent should be.

### Acceptance Criteria

- [ ] When Single Player mode is selected, exactly one editable name field is displayed for the human player.
- [ ] The human player name field is pre-filled with the default value "Jugador 1".
- [ ] The AI opponent's name is displayed in a read-only area — it is not an editable field.
- [ ] An AI difficulty selector is displayed with three options: Easy, Medium, and Hard.
- [ ] "Easy" is selected by default when Single Player mode is first activated.
- [ ] If the player changes the difficulty and then switches modes and back, the difficulty resets to its default ("Easy").
- [ ] The AI difficulty selector is not visible when Multiplayer mode is active.

---

## US-4: Configure a Local Multiplayer Session

**As a** group of players wanting to share a device,  
**I want** to choose how many players are joining and enter all their names,  
**so that** the game engine creates the correct number of player entities for the session.

### Acceptance Criteria

- [ ] When Multiplayer mode is selected, a player-count selector is displayed with options 2, 3, and 4.
- [ ] The default selected count is 2 when Multiplayer mode is first activated.
- [ ] Exactly as many name fields are displayed as the currently selected player count.
- [ ] Each name field is pre-filled with the default value "Jugador N" (where N matches the player's position, starting at 1).
- [ ] Increasing the player count adds a new name field at the end with the appropriate default name.
- [ ] Decreasing the player count removes the last name field; names already entered in the remaining fields are preserved.
- [ ] The player-count selector is not visible when Single Player mode is active.
- [ ] The AI difficulty selector is not visible when Multiplayer mode is active.

---

## US-5: Default Player Names Prevent Empty-Form Blocking

**As a** player who does not want to type a name,  
**I want** the name fields to already have sensible default values,  
**so that** I can start the game immediately without filling in any field manually.

### Acceptance Criteria

- [ ] All player name fields are pre-filled with their default values ("Jugador-1", "Jugador-2", etc.) on initial load and whenever fields are generated by a mode or count change.
- [ ] The Play button is enabled immediately on page load without the player needing to interact with any field.
- [ ] Default values are replaced character-by-character as the player types — the input is a standard editable text field.
- [ ] If a player clears a field and re-focuses it without typing anything, the default value is NOT automatically restored — the field stays empty and shows a validation error.

---

## US-6: Validate Player Names Before Starting

**As a** developer and as a player,  
**I want** the form to prevent starting a game with empty player names,  
**so that** the game engine always receives a valid list of named players.

### Acceptance Criteria

- [ ] If a player manually clears a name field, a visible validation error message appears on or below that field.
- [ ] The error message is in Spanish and clearly states that a name is required.
- [ ] The Play button is disabled while any required name field is empty.
- [ ] The Play button becomes enabled again as soon as all empty name fields are filled.
- [ ] Validation errors are surfaced on blur (when the field loses focus) — errors do not appear while the player is actively typing.
- [ ] Validation error messages are programmatically associated with their input field so screen readers announce them.

---

## US-7: Start the Game Session

**As a** player who has finished configuring the session,  
**I want** to press a "Jugar" button and be taken to the game,  
**so that** the session starts with the configuration I entered.

### Acceptance Criteria

- [ ] A clearly labeled "Jugar" (Play) button is visible on the setup panel.
- [ ] The button is visually distinct and prominent — it is the most obvious call-to-action on the screen.
- [ ] Pressing "Jugar" when the form is valid collects all form values (mode, player names, AI difficulty if applicable) into a game-configuration object.
- [ ] The game-configuration object is stored in a shared application-level service before navigation occurs.
- [ ] The application navigates to the game board route immediately after storing the configuration.
- [ ] If the form is invalid (any required field empty), pressing "Jugar" does nothing and the button remains visually disabled.
- [ ] The button responds to the press within 100 milliseconds with visible feedback (e.g., active/pressed state).

---

## US-8: Use the Lobby on a Mobile Device

**As a** player on a smartphone,  
**I want** the Lobby to be fully usable on a small screen,  
**so that** I can set up and start a game comfortably without pinching or horizontal scrolling.

### Acceptance Criteria

- [ ] The Lobby layout has no horizontal overflow on viewports from 320 px wide upward.
- [ ] All interactive elements (inputs, selectors, the Play button) are large enough to tap comfortably with a finger (minimum 44 × 44 CSS pixels touch target).
- [ ] The hero/branding area and setup panel stack vertically on narrow viewports.
- [ ] The keyboard does not cover the active input field without the layout providing a way to scroll to it.
- [ ] On tablet and desktop viewports, the layout uses the additional space meaningfully (e.g., side-by-side arrangement, larger decorative elements).

---

## US-9: Navigate the Lobby with Keyboard Alone

**As a** player who uses a keyboard for navigation,  
**I want** to complete the entire Lobby flow without touching a mouse or trackpad,  
**so that** the application is accessible to users with motor impairments or keyboard preferences.

### Acceptance Criteria

- [ ] All interactive controls (game mode selector, player-count selector, name inputs, difficulty selector, Play button) are reachable by pressing Tab in a logical order.
- [ ] Focus order follows the visual reading order of the page (top-to-bottom, left-to-right).
- [ ] Each focused control has a clearly visible focus ring or highlight.
- [ ] The game mode selector and difficulty selector are operable with arrow keys when focused.
- [ ] Pressing Enter or Space on the Play button submits the form (when valid).
- [ ] Pressing Enter while focused on a name input does not accidentally submit the form if other fields are empty.
